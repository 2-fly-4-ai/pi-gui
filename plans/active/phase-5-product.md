# Phase 5 — Product Maturity (Auto-Update, Crash Reporting, Onboarding, Platforms)

## Goal

Close the gaps between "works for us" and a Codex-class beta product: users stay current automatically, failures are heard about, first run succeeds unaided, and platform coverage grows deliberately.

## Findings being fixed

- Auto-update was previously a release-page-only check; local updater integration now exists, but the signed staging round-trip still needs a real release pair.
- No crash/error reporting; observability is local-only.
- First-run relies on the README ("go to Settings > Providers"); `model-onboarding-notice` exists but isn't a guided flow.
- Platforms: macOS arm64 + Linux AppImage only; no Windows, no mac x64.
- `apps/website` was a bare Next.js stub; `docs/`+`plans/` needed an index.

## Constraints

- Signed/notarized release pipeline and Homebrew sync scripts already exist — auto-update must fit that pipeline, not replace it.
- Crash reporting must be opt-in and privacy-explicit (this app sees users' code and transcripts; never ship transcript content in reports).
- Windows is a large sub-project — scope it as investigate → spike → decide, not a blind commitment inside this phase.

## Workstreams (independently shippable, in value order)

### 5a. Auto-update
- `electron-updater` against GitHub Releases for the DMG path; verify signature/notarization compatibility with the existing `release.yml` + `release-homebrew-sync` flow.
- In-app UX: unobtrusive "update ready — restart" affordance; respect Homebrew installs (detect and defer to `brew upgrade` messaging instead of self-updating).
- Success: mac user on version N gets N+1 installed and running without visiting the releases page. Verified with a real signed release pair (staging tag).
- Implemented 2026-07-09 local integration slice: packaged direct installs now use `electron-updater` with the existing GitHub Releases publisher metadata, auto-download updates, publish update status through main/preload/renderer IPC, and expose a compact topbar action for downloaded updates (`Restart to update`). Homebrew-managed installs are detected and never self-update; they show `Run brew upgrade`, copy `brew upgrade --cask pi-gui`, and the manual Check for Updates menu explains the Homebrew path. Non-packaged/non-updater builds keep the release-page fallback.

### 5b. Crash & error reporting
- Wire Electron `crashReporter` (native crashes) plus main/renderer uncaught-exception hooks into the existing observability service.
- Opt-in dialog on first run; reports contain stack + app/OS version only. Destination: start with "prefill a GitHub issue from the diagnostics panel" (zero infra), leave a hosted sink (Sentry) as a follow-up decision.
- Success: a forced test crash produces an actionable report artifact; nothing is sent without opt-in.
- Implemented 2026-07-09 first zero-infra reporting slice: App Logs now has a Settings > General opt-in (`Enable diagnostic issue drafts`) and a `Draft issue` action that opens a GitHub issue draft with app/runtime versions plus redacted diagnostic event metadata. The draft excludes raw payloads, source paths, transcripts, prompts, file contents, and obvious secrets; nothing is sent automatically.
- Implemented 2026-07-09 native crash artifact slice: Settings > General now has a separate `Enable local native crash reports` opt-in. Persisted opt-in starts Electron `crashReporter` before windows are created with `uploadToServer: false`; runtime opt-in starts it for the current/future process lifetime. App Logs includes local Crashpad artifacts only when the native opt-in is enabled, and the issue-draft path still sends nothing automatically.
- Implemented 2026-07-09 first-run reporting opt-in UX: New Thread now shows a compact first-run diagnostics prompt until the user chooses `Enable diagnostics` or `Not now`. Enabling turns on redacted issue drafts plus local native crash artifacts, records the decision, and still sends nothing automatically; choosing in Settings also marks the first-run prompt complete.
- Implemented 2026-07-09 packaged forced-crash proof: the production lane now opts into diagnostics, forces a real packaged-app main-process `process.crash()`, relaunches the same profile, and verifies App Logs surfaces the resulting Crashpad artifact.

### 5c. First-run onboarding
- Guided flow when no provider auth exists: detect state (extend `deriveModelOnboardingState`), walk through provider OAuth/API-key setup, land in a ready composer with a suggested first prompt.
- Success: fresh user with zero config reaches a working first session without reading the README. Verified as a packaged-app e2e (`tests/production/`) with a clean `PI_APP_USER_DATA_DIR`.
- Implemented 2026-07-09: clean no-provider profiles now see a compact first-run guide in New thread with a provider-setup action and starter prompt action. Settings remembers the primary surface that opened it, so the provider setup path returns to New thread instead of dropping the user at the empty workspace screen. After saving an API key, the guide disappears, the starter prompt remains, the user can choose a model, and the first session can start without consulting the README.

### 5d. Platform expansion
- Investigate: mac x64 build (cheap — add arch to electron-builder matrix, verify `node-pty` + notification helper universal/dual builds); Windows port (hotspots: `node-pty` on Windows/ConPTY, notification helper is macOS-specific, path handling, menu/shortcut conventions).
- Deliverable of this workstream: a written go/no-go with effort estimate per platform; mac x64 likely ships, Windows becomes its own plan if go.
- Implemented 2026-07-09: recorded the evidence-backed platform decision in `docs/platform-expansion.md`. mac x64 is a go for a packaging spike, but not shipped until helper packaging, packaged-runtime verification, release assets, Homebrew policy, and x64 smoke coverage are complete. Windows is a no-go for direct Phase 5 shipment and should become a dedicated platform plan after a runtime/terminal/packaging spike.

### 5e. Website & docs
- Either build a minimal real landing page in `apps/website` (screenshots, install instructions, link to releases) or remove the package until it's wanted.
- Add `plans/README.md` index marking active vs archived; keep `docs/architecture.md` (phase 2) current.
- Implemented 2026-07-09: added `plans/README.md` with active-plan links and standalone/archived plan status. Remaining 5e work: decide whether `apps/website` becomes an install/docs surface or is removed/deferred.
- Implemented 2026-07-09: kept `apps/website` as the product install/docs surface. It is now a static-export Next site with metadata, sitemap/robots, GitHub Releases install path, Homebrew/macOS/Linux setup notes, architecture links, and demo media. Verified with `pnpm --filter @pi-gui/website run build` (static export passed; Next emitted only the existing ESLint-plugin notice).

## Success Criteria

- Auto-update has local updater integration and Homebrew non-self-update guidance. Remaining closeout: a verified signed staging round-trip with real version N/N+1 release artifacts.
- Crash/error reporting is opt-in, privacy-reviewed, and sends no transcript, prompt, file-content, or secret material. Status: issue-draft opt-in, local native crash artifact opt-in, first-run reporting opt-in UX, and packaged forced-crash proof implemented.
- A clean-profile user can reach a working first session through guided onboarding without consulting the README. Status: implemented with packaged-app proof.
- Platform expansion has an evidence-backed decision: mac x64 shipped or explicitly deferred; Windows split into its own plan if the spike says go. Status: decision recorded; mac x64 packaging spike is approved, Windows direct shipment deferred.
- `apps/website` is a useful install/docs surface or intentionally removed from the workspace. Status: implemented as a static install/docs surface.
