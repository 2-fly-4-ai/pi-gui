# Agents & Subagents — From Text-Scraping to a Real Orchestration Surface

## Goal

Make agent definitions and subagent runs first-class product features: typed lifecycle instead of regex over raw text, live visibility instead of fire-and-forget, durable run records instead of in-memory lists. This is the agent-side twin of phase 4 (same disease: unstructured data over a boundary, patched with string surgery).

## How it works today (verified 2026-07-08)

1. **Agent definitions** (`apps/desktop/electron/agent-definitions.ts` + editor UI): a genuinely solid layer — builtin configs for ~15 canonical roles, global (`~/.pi/agents`) and project (`.pi/agents/*.md`) overrides, frontmatter parsing with warnings, plus the shinobi roster theming (avatars/names per role).
2. **Subagent execution** happens entirely inside the upstream `pi` runtime's Agent tool. The driver event union (`packages/session-driver/src/types.ts`) has **no subagent events** — the desktop only sees generic `toolStarted/Updated/Finished` for the Agent call.
3. **Results come back as raw text.** `sanitizeToolOutputText` in `apps/desktop/electron/app-store-timeline.ts:502` regex-detects JSONL transcript dumps and reconstructs a summary by scraping `Agent:/Type:/Description:` lines and a `STATUS:` block with brittle string surgery (`.replace(/"?\}\]?[,]?$/, "")`-grade). When the heuristics miss, the chat gets megabytes of raw JSONL — `agentlog.txt` (19.6 MB, "Tool uses: undefined", "Live session controls are unavailable") is a captured example of that failure class.
4. **Workflows are fire-and-forget prompt injection.** `SubagentRunStore` (`apps/desktop/electron/subagent-runs.ts`) holds runs in a plain in-memory array (lost on restart), status is only `submitted | failed` (failed = the *submission* failed), and "running a workflow" means appending a `SUBAGENT_WORKFLOW_RUN` marker prompt that asks the model to "use the Agent(...) tool when appropriate" — nothing tracks whether it did, finished, or produced the promised artifacts. The timeline card then re-parses that marker text with regex (`subagent-timeline-card.ts`).
5. **Observability panel** tails log files (desktop log, ledger, `subagents-audit.jsonl`, session JSONL) and classifies lines with heuristics (`event.includes("tool") ? "tool" : "subagent"`). It's a log viewer, not live state.
6. **Role sprawl.** 15+ canonical roles plus legacy aliases; workflow templates reference pseudo-roles (`reviewer/correctness`) that aren't definitions; nothing validates template roles against existing definitions.

## Constraints

- Keep `pi-sdk-driver` thin over `pi-mono`; prefer consuming what the runtime already emits (the subagents-audit JSONL, tool events) over forking runtime behavior. Where the runtime's output is genuinely unstructured, push a structured-output request upstream and build a versioned parser as the interim, not scattered regexes.
- Session correctness rules apply: subagent cards must survive thread return, restart, and re-hydration.
- Coordinate with phase 4: subagent lifecycle events should ride the same typed event protocol; don't build a second ad-hoc channel.
- Local branches `feature/nico-lite-subagents` and `feature/agent-observability-panel` contain prior iterations — mine for findings, don't resurrect wholesale.

## Workstreams

### W1. Typed subagent lifecycle (the root fix)

- Completed 2026-07-09 inventory of what the checkout consumes today:
  - `SessionSupervisor.mapAgentEvent` turns upstream `tool_execution_start/update/end` into generic `toolStarted/Updated/Finished` driver events. Agent invocations are only distinguishable by `toolName === "agent"`/`.agent` and `input.subagent_type`/`subagentType`; there is no driver-level subagent identity, status, transcript path, elapsed time, artifact list, or child tool count.
  - The typed event surface lives in `packages/session-driver/src/types.ts` and is mirrored by `packages/pi-sdk-driver/src/vendor/session-driver.d.ts`; any W1 event addition must update both or the package boundary drifts.
  - `subagents-audit.jsonl` is already tailed by `ObservabilityService` from `PI_SUBAGENTS_AUDIT_LOG` or `<agentDir>/logs/subagents-audit.jsonl`. Known consumed fields are `ts`/`timestamp`, `event`, `agentId`/`id`, `type`, `description`, `status`, `toolName`, `toolCallId`, `args`, `path`, `cwd`, `workspaceRoot`, `reason`, `error`, `resultExcerpt`, `isError`, and `durationMs`. Known event names in current tests/normalizer are `subagent_spawn_blocked`, `subagent_tool_blocked`, `subagent_tool_start`, `subagent_tool_end`, `subagent_failed`, and `subagent_completed`.
  - Transcript `.output` references are currently discovered only from raw Agent tool output text via `/.../tasks/<id>.output`; the timeline sanitizer summarizes JSONL-looking dumps with `pi-subagent-result-v1` as a fallback, but that path cannot produce live progress.
  - `SubagentRunStore` persists the workflow submission record, but it is not correlated to the eventual Agent tool call because the prompt marker has no structured correlation id that returns through the driver event stream.
- Add `SubagentRunUpdated` to the driver event union: `{ runId, parentSession, role/agentName, status: started|progress|completed|failed|cancelled, toolUseCount, elapsedMs, summary?, transcriptPath?, artifacts? }`, sourced by ingesting the audit stream in the driver (thin adapter, no runtime fork).
- W1 implementation boundary: add a correlation id to workflow prompts and persisted run records; create a thin audit adapter in the driver/main-process layer that folds audit lines plus Agent tool events into `SubagentRunUpdated`; keep `ObservabilityService` as a historical/log viewer over the same source, not the owner of live state.
- Implemented 2026-07-09 first slice: `SubagentRunUpdatedEvent` is now part of the shared `SessionDriverEvent` union and vendored driver mirror. `SessionSupervisor` emits `started/progress/completed/failed` lifecycle events from Agent tool start/update/end using the Agent tool call id as `subagentRunId`; the timeline consumes the event into a stable Agent row with expandable summary, tool-use count, elapsed time, transcript path, and artifact paths when those fields are present. Remaining W1 work: audit-stream ingestion/backfill, runtime/audit correlation to the prompt-level workflow id, terminal `cancelled` mapping, and live artifact backfill from audit/watch sources.
- Implemented 2026-07-09 prompt-correlation slice: `SubagentRunStore` now creates the durable run id before submission, stores it as `workflowRunId`, injects it into the `SUBAGENT_WORKFLOW_RUN` marker as `workflow_run_id`, and the timeline marker parser carries that id through to the card. This gives later audit adapters, queued-run checks, and cancel controls a stable workflow handle without relying only on role/session matching.
- Timeline consumes the event instead of scraping tool output. `sanitizeToolOutputText` and its regex family get deleted, replaced by one versioned result-parser used only as fallback for runtimes without audit events.

### W2. Structured result contract

- The `STATUS: DONE|BLOCKED|...` vocabulary currently lives implicitly in regex. Move it into the builtin agent definitions' prompts (single source), and parse exactly one versioned final-block format with a validator; on parse failure, show "summary unavailable — open transcript" instead of leaking raw JSONL into chat.
- The definitions layer already supports an `output` frontmatter key — investigate whether the runtime supports structured/JSON final output per agent; if yes, prefer it and drop text parsing for those roles.
- Hard cap tool-output rendering size in the timeline regardless (defense in depth: no multi-MB message can ever hit the transcript again; oversized payloads become a "view full output" affordance).
- Implemented 2026-07-09: transcript ingestion and renderer fallback now cap visible tool-output text at 12,000 characters with an explicit truncation note. This covers both generic tool output and summarized subagent transcript output, and prevents missed heuristics from injecting multi-MB payloads into chat.
- Implemented 2026-07-09: builtin agent definitions now append a shared `pi-subagent-result-v1` final response contract, and the transcript sanitizer uses a validator for that versioned block instead of scraping loose `STATUS:` prose. Missing/invalid final blocks render as "Summary unavailable - open transcript" with the raw JSONL still hidden from chat.
- Investigated 2026-07-09: the repo-local `output` frontmatter is parsed, serialized, and displayed by the desktop definitions layer, but no path was found that maps it to a runtime response-format, JSON-schema, or structured-output setting in `packages/pi-sdk-driver` or the desktop runtime bridge. Keep the versioned text block as the current contract until upstream/runtime support exists.
- Implemented 2026-07-09: summarized subagent result rows now preserve the extracted `.output` path as structured timeline metadata and render a collapsed "Full output" affordance with a copy-path button. The raw JSONL/full payload remains hidden from chat and is not stored in renderer state as an uncapped transcript body.

- Implemented 2026-07-09: active `submitted`/`running` workflow records in Settings > Subagents > Runs now expose a cancel action. Cancellation uses the stored target session, persists the workflow as `cancelled` only after the existing session cancel path returns, stops live artifact scanning for the terminal run, and leaves retry/open-transcript actions available on the cancelled record. Remaining W3 work: runtime/audit event correlation to `workflowRunId` and multi-role/multi-child aggregation.

- The workflow marker (`SUBAGENT_WORKFLOW_RUN` text blob) is replaced by structured message metadata carried through the driver, so the card renders from data, not regex over the prompt.
- Display Mode tiles show subagent activity for their thread (n running, roles) so parallel-run supervision actually works from the command center.
- Implemented 2026-07-09 first Display Mode slice: command-center tiles now carry a full-transcript-backed summary of typed Agent lifecycle rows with a quiet activity strip such as `1 Agent running · reviewer`, including completed/failed fallback states even when the Agent row is outside the tile's clipped recent transcript.
- Implemented 2026-07-09 W4 durable-run backfill slice: Display Mode now merges transcript-derived Agent activity with persisted workflow run records for the same session, so command-center tiles still show active/failed/completed workflow activity when the transcript has no Agent lifecycle row cached.
- Implemented 2026-07-09 W4 structured workflow-message metadata slice: workflow submissions attach typed `subagent-workflow` metadata to the transcript/queued message path, timeline cards render from metadata first, and the old `SUBAGENT_WORKFLOW_RUN` text parser remains only as a compatibility fallback for older persisted transcripts. Remaining W4 work: add the full transcript side-drawer path.

### W5. Workflow engine honesty

- Templates today *hope* the model uses the Agent tool. Measure it (W1 events tell you); where the runtime supports it, constrain the orchestration turn (tool choice / explicit delegate agent) instead of politely asking.
- Validate template roles against existing definitions at load; a template referencing a missing role surfaces a warning with a one-click "create from builtin" fix.
- Implemented 2026-07-09: built-in workflow cards now validate their roles against the enabled agent definitions before launch. Slash-qualified workflow variants such as `reviewer/tests` resolve through the base `reviewer` role, disabled or missing roles produce an inline warning, and invalid workflow launch buttons are disabled instead of silently submitting a prompt that cannot delegate correctly.
- Implemented 2026-07-09: workflow warnings for disabled built-in roles now include a one-click restore action that resets the visible override back to the built-in definition, removes the warning, and re-enables launch without leaving the Workflows tab.
- Implemented 2026-07-09: direct workflow turns that finish without any matching Agent lifecycle event now move from `submitted` to `failed` with a visible "Workflow finished without invoking the Agent tool." message instead of staying submitted forever. Queued follow-up workflows now have a durable prompt-level `workflow_run_id`; exact no-Agent measurement still needs the parent-turn/runtime correlation that connects queued execution completion back to that id.
- Allow user-defined workflow templates (persisted next to agent definitions) — the six builtins become data, not code.

### W6. Definitions editor & role hygiene

## Sequencing

1. W1 inventory + event design (gates everything; do the audit-schema inventory first).
2. W2 output cap + versioned parser (immediate user-visible fix for raw-JSONL leaks — can ship before the event work).
3. W3 persistence + lifecycle, then W4 cards.
4. W5–W7 in any order after.

Coordinate W1/W4 with phase 4's protocol work; the transcript-size cap in W2 and run persistence in W3 have no dependencies and can start now.

## Success Criteria

- No raw subagent JSONL can appear in a transcript (fault-injection test: feed a multi-MB JSONL tool result through the timeline; renders as capped card with working "open transcript").
- Subagent runs show live status, elapsed, and tool-use counts sourced from events, not text; counts are never `undefined`.
- Runs survive app restart with correct terminal status; reopened sessions show completed run cards with summaries and artifacts.
- A workflow run whose model never invoked the Agent tool is visibly reported as such (not silently "submitted" forever).
- Template/role validation: referencing a missing agent definition is a visible warning, not a silent no-op.
- All existing subagent-related e2e specs pass; new specs cover the lifecycle card (start → progress → complete) and the raw-JSONL fault injection.
