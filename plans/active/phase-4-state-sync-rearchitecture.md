# Phase 4 — State Sync Rearchitecture (Delta IPC + Streaming Transcripts)

## Goal

Replace the full-snapshot state model with delta-based sync so streaming cost is O(change), not O(total state / transcript length). This is the root fix for long-session chat performance — a product feature per repo rules, not polish.

## Findings being fixed

- Every store mutation returns and broadcasts the entire `DesktopAppState`; `structuredClone` on each read; 250ms coalesced publisher.
- The selected transcript is re-sent wholesale on every batched assistant delta — O(full transcript) serialization per tick while streaming.
- Every preload method resolves with the full `DesktopAppState`, so each user action also pays full-state serialization.
- Prior perf work exists in stale worktrees (`chat-performance-foundation`, `layout-perf-diagnostics`); mine them for findings before designing (read-only — do not resurrect their code wholesale).

## Current surfaces to change

- Store emitters: `apps/desktop/electron/app-store.ts` (`emit`, `publishSelectedTranscript`, `buildSelectedTranscriptRecord`) and the helper modules under `apps/desktop/electron/app-store-*.ts` that currently return `Promise<DesktopAppState>`.
- IPC publishers: `apps/desktop/electron/main.ts` (`attachStatePublisher`, `createCoalescedIpcPublisher`, `shouldPublishStateImmediately`).
- Preload/API contract: `apps/desktop/electron/preload.ts` and `apps/desktop/src/ipc.ts` where most mutating methods currently resolve with `DesktopAppState`.
- Renderer subscription/materialization: phase 3's planned `apps/desktop/src/state/` store, replacing `useDesktopAppState`/`updateSnapshot` in `apps/desktop/src/App.tsx`.
- Diagnostics hooks: `apps/desktop/src/timeline-diagnostics.ts`, `apps/desktop/src/renderer-diagnostics.ts`, and `apps/desktop/electron/diagnostics.ts`.

## Constraints

- Clean reimplementation of the sync layer, not patches around it (root AGENTS.md).
- Session correctness is non-negotiable: no lost/duplicated/reordered timeline items across steering, cancels, session switches, thread-return hydration, or app restart.
- The store's internal model can stay; the change is the transport + renderer subscription contract (the phase 3 `src/state/` seam).
- `AssistantDeltaBatcher` and `createCoalescedIpcPublisher` remain useful concepts — rebuild on them where they fit.
- Keep `pi-sdk-driver` thin; no runtime behavior forks.

## Design direction (validate before building)

1. **Transcript channel → append-only event stream.** Main sends `transcriptReset(sessionRef, snapshot)` on selection/hydration, then `transcriptAppend`/`transcriptUpdateLast` (batched deltas), `transcriptTruncate` for tree navigation. Renderer keeps the materialized array; a monotonic sequence number per session detects gaps → renderer requests a reset (self-healing).
2. **App-state channel → scoped patches.** Split `DesktopAppState` into domains (workspaces, sessions list, composer, runtime jobs, settings, notifications...). Publish per-domain replacement objects (simpler than JSON-patch, still O(domain)); renderer store from phase 3 fans out to selector subscribers.
3. **Mutations stop returning full state.** Preload methods return `void`/small results; the state channel is the single source of updates. This removes the other full-serialization path and simplifies ~100 preload signatures.
4. Keep a `getState()` full-snapshot request for boot + resync.

## Protocol decisions to make before coding

- Define shared event types in `apps/desktop/src/ipc.ts` first, then implement main/preload/renderer from those types so the contract stays compile-checked.
- Each transcript event carries `{ workspaceId, sessionId, sequence }`; reset carries `{ sequence, transcript }`; append/update/truncate carry only the changed payload. Sequence scope is per session, not global app state.
- Renderer ignores transcript events for non-selected sessions unless/until the state store supports background transcript caches. State patches still carry enough selection metadata for a later `getSelectedTranscript()` reset.
- Gap handling is deterministic: if `incoming.sequence !== expected + 1`, renderer calls a reset request once, marks the transcript as resyncing, and suppresses partial rendering until the reset lands.
- Truncate semantics must be explicit for tree navigation: truncate by stable timeline item id when available; use an index only as a fallback after reset, because indexes drift during streaming.
- Keep the legacy full-snapshot channels behind a short-lived compatibility flag only while both publisher paths exist. The flag must be removed before the phase closes.

## App-state domains to validate

Start with replacement-per-domain messages, then split further only if instrumentation proves a domain is still too large:

- `selection`: selected workspace/session/view/sidebar/display-mode flags.
- `workspaces`: workspace list, worktree records, archive/session summaries.
- `composer`: draft, attachments, queued messages, draft-sync source/nonce.
- `runtime`: running jobs, provider/model/settings readiness, host UI requests.
- `settings`: model scope, skill/extension settings, notification prefs, custom instructions, shell.
- `diagnostics`: last error, notifications, observability counters that affect UI.

Composer migrates last because `composerDraftSyncSource` currently controls immediate state publication and draft overwrite behavior.

- 2026-07-08 — `feature/layout-perf-diagnostics` (`.worktrees/layout-perf-diagnostics`, HEAD `c7f23e6`): useful conclusions are renderer timeline diagnostics behind `PI_APP_LAYOUT_MONITOR` / `PI_APP_PERF_MONITOR` (layout-shift, row-resize, scroll-frame, composer-resize, long-task events), plus tests covering long-message virtualization, thread-return hydration, native-scroll intent, and scroll/pinning stability while streaming. Current main already contains the diagnostic hook and most of this hardening. Treat the branch as measurement/test inspiration only; it mixes broad UI/subagent/product changes with perf work and is not a clean implementation source.
- Shared conclusion from both branches: batching and virtualization reduce visible jank, but the transport still sends full `DesktopAppState` snapshots and full selected-transcript payloads (`buildSelectedTranscriptRecord` clones the whole transcript; mutating APIs still resolve with full state). Phase 4 should therefore keep the existing diagnostics as the before/after measurement harness and replace the transport contract with typed transcript events + state-domain patches rather than extending the current full-payload publisher.

## Success Criteria

- Streaming a reply into a 1k+ message transcript serializes only the delta per tick (measured: >100x reduction in bytes/sec on the transcript channel vs baseline).
- No dropped/duplicated timeline items under: rapid steering, cancel mid-stream, session switch mid-stream, tree navigation, app restart mid-stream, renderer reload (gap-detection resync proven by a fault-injection test).
- Mutating preload APIs no longer return `DesktopAppState` except where a small domain-specific result is required; `getState()` remains only for boot/resync/test helpers.
- All core + live e2e specs pass; `thread-return-hydration`, `queued-messages`, `composer-draft-sync`, and `timeline-native-scroll` specs get extended for the new protocol edges.
- UI latency for user actions (send, switch session) does not regress (before/after timings).
